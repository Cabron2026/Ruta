# Ruta — desktop + mobile, единый проект

## Структура

```
index.html     — роутер: определяет устройство и редиректит на нужную версию
desktop.html   — десктоп-версия (без изменений, эталон MacBook 13" / Chrome)
mobile.html    — мобильная версия (точечно доработан safe-area / dvh,
                  эталон iPhone 16 Plus / Chrome, подтверждённо работает в Telegram)
```

Это три полностью самостоятельных статических файла. Никакой сборки, никакого
сервера — просто кладёшь файлы в репозиторий и включаешь GitHub Pages.

## Как включить GitHub Pages

1. Создай новый репозиторий на GitHub (или используй существующий)
2. Положи в его корень эти три файла (`index.html`, `desktop.html`, `mobile.html`)
   и закоммить
3. Зайди в репозиторий → **Settings → Pages**
4. В разделе **Build and deployment** выбери:
   - Source: **Deploy from a branch**
   - Branch: **main** (или твоя основная ветка), папка **/ (root)**
5. Сохрани — через 1-2 минуты GitHub покажет ссылку вида:
   `https://<твой-username>.github.io/<имя-репозитория>/`

## Итоговые ссылки

После включения Pages у тебя будет:

- **Главная (роутер, авто-определение устройства):**
  `https://<username>.github.io/<repo>/`

- **Прямая ссылка на мобильную версию** (для тестов в Telegram, в обход роутера):
  `https://<username>.github.io/<repo>/mobile.html`

- **Прямая ссылка на десктоп-версию:**
  `https://<username>.github.io/<repo>/desktop.html`

Ссылку на `mobile.html` удобно сохранить отдельно — именно её стоит кидать себе
в Telegram для тестирования после любых правок, не дожидаясь, пока роутер
определит устройство.

## Как обновлять после правок

Любое изменение — это просто:

```
git add .
git commit -m "описание правки"
git push
```

GitHub Pages подхватывает изменения автоматически, обычно в течение
1-2 минут после пуша. Обновлять ничего вручную на стороне GitHub не нужно.

## Логика роутинга (index.html)

Кратко, в порядке приоритета:

1. Если страница открыта внутри Telegram (Mini App / in-app browser) —
   всегда показывается `mobile.html`, без исключений.
2. Если устройство touch-first (`pointer: coarse`) — то:
   - в портретной ориентации → `mobile.html`
   - в альбомной ориентации (включая большие тач-мониторы) → `desktop.html`
3. Иначе (мышь/трекпад как основной ввод) → `desktop.html`

Редирект происходит один раз при заходе и не реагирует на последующий ресайз
окна браузера — версия не "прыгает", если на десктопе вручную потянуть окно.

## Что было точечно доработано в mobile.html

Чтобы не задеть анимации и JS-логику скролла (которая уже подтверждённо
работает на iPhone 16 Plus и в Telegram), внесены только два аддитивных
изменения:

1. Корневой контейнер: `min-height: 100vh` → `min-height: 100dvh`
   (устраняет потенциальный "прыжок" высоты при скрытии/появлении адресной
   строки браузера на мобильных; на iPhone 16 Plus в Safari/Chrome визуально
   не отличается от исходного поведения).
2. `env(safe-area-inset-*)` добавлены аддитивно (через `max()`, не заменяя
   существующий padding) в хедер (сверху/по бокам) и в нижнюю строку футера
   (снизу) — защищает контент от наезда на вырез/динамический остров и
   home indicator на устройствах с safe area. На устройствах без выреза
   `env()` возвращает 0, так что внешний вид не меняется.

`desktop.html` не изменялся: используемые там `100vh` не подвержены проблеме
"плавающей" адресной строки (на десктопе её нет), а вёрстка уже построена на
текучих `clamp()`/`rem` единицах.
